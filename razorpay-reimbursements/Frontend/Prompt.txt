Build a frontend for a reimbursements management tool. This is a Vanilla JS + HTML + CSS project — no frameworks, no libraries except what is explicitly listed below. The backend is already running at http://localhost:7002 and uses cookie-based auth (httpOnly JWT cookie named token).
The most important rule: this must not look AI-generated or vibecoded. Every design decision must feel intentional — like a real product designer made it. No generic Bootstrap cards, no centered emoji headers, no rainbow gradients, no shadow-on-everything. Think Vercel, Railway, Linear — dark surfaces, tight typography, purposeful whitespace.

Folder structure — create every file:
frontend/
  index.html          ← login/register page (public)
  dashboard.html      ← protected, role-aware dashboard shell
  css/
    reset.css         ← minimal CSS reset

    dashboard.css     ← dashboard-specific styles

Design system — implement these exactly:
Typography:

Font: Inter from Google Fonts. Import it in the HTML head.
Base size: 14px, line-height 1.6
Headings: 500 weight only — never 700 or 800
Labels, metadata: 12px, letter-spacing 0.04em, uppercase, muted color

Theme toggle:

A sun/moon icon button in the top-right of the header on every page
Clicking it adds/removes the light class on <html>
Persists to localStorage as key theme
On page load, read localStorage and apply the saved theme before render — no flash


Auth page (index.html):
Two panels side by side on desktop — left panel is a full-height dark surface with the product name "Reimbursements" in large 500-weight text, a one-line tagline below it in muted color, and a subtle grid or dot pattern background (CSS only, no images). Right panel is the form area.
The form area has two tabs — "Sign in" and "Register" — rendered as plain text links that swap the visible form, not browser tabs. No rounded pill tabs, no colored tab backgrounds. Just an underline on the active one.
Sign in form fields: email, password. Register form fields: name, email, password. Every input has a floating label or a clean above-label (not placeholder-as-label). Inputs have a 1px border using --border, background --bg-elevated, no box-shadow. On focus, border color transitions to --accent with a 150ms ease transition. No glow, no shadow on focus.
Submit button: full width, background --accent, 500 weight, 14px, 40px height, --radius-md. On hover background transitions to --accent-hover. On loading state, button text changes to a subtle animated ellipsis (Signing in...) and is disabled. No spinner icon.
Error messages appear inline below the relevant field in --danger color, 12px, no background, no border — just plain text.
After successful login, redirect to dashboard.html.

Dashboard shell (dashboard.html):
On load, dashboard.js calls GET /rest/employees (or a lightweight profile check) to determine the logged-in user's role. If the cookie is missing or expired, redirect to index.html immediately.
Layout: fixed sidebar on the left (220px wide), main content area fills the rest. Header is a 52px bar at the top of the main area — not full-width, it sits inside the content column. Sidebar and content area both use --bg-surface. The page background (--bg-base) is visible only as a 1px gap between sidebar and content — a visual separator without a hard border.
Sidebar:

Top: product logo or wordmark ("Reimb.") in 500 weight 15px, left-padded 20px, vertically centered in a 52px header area matching the content header height
Middle: nav links. Each link is 36px tall, full-width, left-padded 20px, 14px, --text-secondary color. Active link: --text-primary color, background --bg-hover, left border 2px solid --accent. Hover: background --bg-hover, color --text-primary. No icons — text only unless you can use a simple unicode character (→, ·) as a subtle indicator.
Bottom: user info section — name in 13px --text-primary, role badge, and a "Sign out" link in --text-secondary 12px

Role badge: inline pill, 11px uppercase, letter-spacing 0.06em, 4px vertical padding, 8px horizontal padding, --radius-sm. Colors by role: EMP → gray (use --text-muted text, --bg-elevated background), RM → accent muted, APE → warning muted, CFO → success muted.
Content header: 52px, contains the current view title on the left (15px, 500 weight) and the theme toggle on the right.

EMP dashboard (views/emp.js):
Nav links: "My Reimbursements", "Submit Request"
"My Reimbursements" view:

Calls GET /rest/reimbursements
Renders a table — not cards. Table has columns: Title, Description, Amount, Status. No checkboxes, no action column.
Status column shows a badge: PENDING → warning colors, APPROVED → success colors, REJECTED → danger colors. Same pill style as role badge.
Empty state: a single centered line of muted text — "No reimbursements yet." No illustration, no big icon.
Table rows have a subtle bottom border using --border-subtle. Row hover: --bg-hover background.

"Submit Request" view:

A single form, max-width 480px, left-aligned (not centered)
Fields: Title (text), Description (textarea, 4 rows), Amount (number)
Same input styling as auth page
Submit calls POST /rest/reimbursements
On success: show a one-line success toast (bottom-right, slides up, auto-dismisses in 3s), then clear the form and switch view to "My Reimbursements"
On error: inline error below the form, same style as auth errors


RM dashboard (views/rm.js):
Nav links: "Pending Approvals", "My Team"
"Pending Approvals" view:

Calls GET /rest/reimbursements
Same table layout as EMP but adds an Actions column with two buttons per row: "Approve" and "Reject"
Approve button: small, outlined style, --success border and text, background transparent. On hover: --success-muted background.
Reject button: same pattern with --danger.
Clicking either opens a minimal inline confirmation — not a modal. The row expands or a small confirmation line appears below it: "Approve this reimbursement? [Confirm] [Cancel]" in 13px muted text.
On confirm: calls PATCH /rest/reimbursements with the correct userId, reimbursementId, status. Refreshes the list on success.

"My Team" view:

Calls GET /rest/employees
Same table: columns Name, Email, Role
No actions — read only


APE dashboard (views/ape.js):
Nav links: "Pending Approvals", "All Employees"
"Pending Approvals" view:

Calls GET /rest/reimbursements — APE sees RIs where rm_approved = true and status is PENDING
Same table + inline approve/reject confirmation as RM

"All Employees" view:

Calls GET /rest/employees
Table: Name, Email, Role — read only


CFO dashboard (views/cfo.js):
Nav links: "Audit Log", "All Employees", "Assign Role", "Assign Manager", "Remove Assignment"
"Audit Log" view:

Calls GET /rest/reimbursements — CFO sees APPROVED and REJECTED
Table: Title, Amount, Status — read only

"All Employees" view:

Calls GET /rest/employees
Table: Name, Email, Role — read only

"Assign Role" view:

Form with two fields: User ID (text input), Role (a plain <select> with options EMP, RM, APE — not CFO)
Calls POST /rest/roles/assign
Success toast on completion

"Assign Manager" view:

Form: Employee ID, Manager ID
Calls POST /rest/employees/assign
Success toast on completion

"Remove Assignment" view:

Form: Employee ID, Manager ID
Calls DELETE /rest/employees/assign
Success toast on completion


api.js — centralize all fetch calls here:
Every function uses fetch with credentials: 'include' so the cookie is sent automatically. All requests set Content-Type: application/json. On any 401 response, redirect to index.html. Export named functions — one per endpoint:
register(name, email, password)
login(email, password)
logout()
assignRole(userId, role)
getEmployees()
assignManager(empId, rmId)
removeAssignment(empId, rmId)
createReimbursement(title, description, amount)
getReimbursements()
getReimbursementsByUser(userId)
updateReimbursement(userId, reimbursementId, status)

utils.js exports:
showToast(message, type) — type is 'success' or 'error'. Toast appears bottom-right, fixed position, 320px wide, --bg-elevated background, left border 3px solid (success color or danger color). Slides up using a CSS keyframe animation (transform: translateY(8px) → translateY(0), opacity 0 → 1, 200ms ease). Auto-dismisses after 3 seconds. Max 3 toasts stacked.
formatAmount(amount) — returns ₹1,500.00 format using toLocaleString('en-IN').
formatDate(isoString) — returns 12 Jun 2026 format.

Things to avoid — these make it look AI-generated:

No box-shadow on cards or inputs (none at all)
No gradient backgrounds anywhere
No border-radius above 12px
No emoji in the UI
No centered hero text with a subtitle below it in the dashboard
No colored section backgrounds (every surface uses the token colors only)
No font-weight: 700 or 800 anywhere
No animation except the toast slide-up and the button loading state
No hover scale transforms (transform: scale(1.02))
No placeholder text that acts as a label


Connectivity:

The backend runs at http://localhost:7002. All API calls in api.js use this base URL. The frontend is served as static files — open index.html directly in a browser or serve it with a simple static server. No build step required.
