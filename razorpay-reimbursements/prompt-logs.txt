2nd Prompt:-

Now that the folder structure and scaffold are in place, implement the business logic for the following modules. Do NOT touch any reimbursement logic yet — leave those controller and service stubs as they are.
Roles & Hierarchy:
There are 4 roles: EMP, RM, APE, CFO. Every new registered user gets EMP by default. The CFO is the only one who can change roles.

AUTH MODULE — implement fully
POST /rest/onboardings/register

Request body: name, email, password
Only allow emails ending in @org.com — reject anything else with 400
Hash password with bcryptjs before storing
Role defaults to EMP automatically — user never picks their role
If email already exists, return 400 with a clear message
On success return { status: 'success', data: { userId, name, email, role } }

POST /rest/onboardings/login

Request body: email, password
Only allow emails ending in @org.com — reject anything else with 400
Verify password against bcrypt hash
On success, sign a JWT containing { userId, email, role } using JWT_SECRET
Set the JWT as an httpOnly cookie named exactly token with sameSite: 'lax'
On success return { status: 'success', data: { userId, name, email, role } }

POST /rest/onboardings/logout

Clears the token cookie
Returns { status: 'success', data: { message: 'Logged out successfully' } }


ROLES MODULE — implement fully
POST /rest/roles/assign

Protected: only CFO can call this (use authenticate + authorize middleware)
Request body: userId, role
Allowed roles to assign: EMP, RM, APE only — assigning CFO role must be blocked with 400
If userId does not exist, return 404
If the role string is not one of the valid values, return 400
If the user already has that role, return 400 with message "User already has this role"
Role downgrade logic: if the user currently has role RM and is being changed to any other role, automatically delete all rows in employee_manager where rm_id matches that user's id. Those employees are now unassigned and can be reassigned to another RM by the CFO later.
On success return { status: 'success', data: { userId, name, email, role } } with the updated user


EMPLOYEES MODULE — implement fully
GET /rest/employees

Protected: EMP is not allowed — return 403 if an EMP calls this
RM: returns only the EMPs assigned to them (rows in employee_manager where rm_id = calling user's id), joined with users table
APE: returns all users with role EMP and all users with role RM
CFO: returns all users (every role including APE and CFO)
Response shape exactly:

json{
  "status": "success",
  "data": {
    "users": [
      { "userId": "...", "name": "...", "email": "...", "role": "..." }
    ]
  }
}
POST /rest/employees/assign

Protected: only CFO can call this
Request body: empId (the EMP's userId), rmId (the RM's userId)
Validate that the user with empId has role EMP — if not, return 400
Validate that the user with rmId has role RM — if not, return 400
If the EMP is already assigned to an RM, return 400 with message "Employee is already assigned to a reporting manager"
On success insert into employee_manager and return { status: 'success', data: { message: 'Employee assigned successfully' } }

DELETE /rest/employees/assign

Protected: only CFO can call this
Request body: empId, rmId
Validate both users exist and have correct roles
If the assignment does not exist, return 404 with message "Assignment not found"
On success delete the row and return { status: 'success', data: { message: 'Assignment removed successfully' } }


MIDDLEWARE — verify these are correctly wired:
authenticate.js — reads JWT from httpOnly cookie named token, verifies with JWT_SECRET, attaches decoded payload to req.user. Returns 401 if cookie missing or token invalid or expired.
authorize.js — factory function, takes array of allowed roles like ['CFO'] or ['CFO', 'RM']. Checks req.user.role. Returns 403 with { status: 'error', message: 'Forbidden' } if role not allowed.
errorHandler.js — catches AppError instances and returns their statusCode and message. For unexpected errors returns 500 with a safe generic message. Never expose stack traces in responses.

EDGE CASES to handle explicitly:

Any protected route called without a cookie → 401
Any route called with a valid cookie but wrong role → 403
POST /rest/roles/assign with role: 'CFO' → 400 with message "Cannot assign CFO role"
POST /rest/employees/assign where empId and rmId are the same user → 400 with message "Employee and manager cannot be the same user"
DELETE /rest/employees/assign where assignment doesn't exist → 404
All DB errors should be caught and passed to the global error handler via next(error)
